# Knowledge/reference/

> Reference materials and domain knowledge. Things you return to, not things you process once.
> External frameworks, guides, standards, competitive landscapes, and domain expertise.

---

## What Goes Here

- Frameworks you apply repeatedly (OKRs, RICE scoring, jobs-to-be-done, etc.)
- Competitive landscape snapshots (keep dated — these go stale)
- Domain expertise notes — what you know about your industry, users, or technology
- Standards and guidelines (design systems, API docs you reference often, compliance notes)
- Reading notes from books or articles that shaped your thinking

---

## File Format

Filename: `[topic].md` — name it for how you'd search for it, not how you found it

```markdown
# [Topic]

**Type:** Framework / Competitive Intel / Domain Knowledge / External Guide
**Last updated:** YYYY-MM-DD
**Source:** [Where this came from]

## Summary

[Why this is useful — what does it help you do or decide?]

## Content

[The actual reference material]

## When to Use

[In what situations should you pull this?]

## Connected Projects

- [[Projects/...]] *(projects where this is relevant)*
```

---

## Maintenance

- Review reference files quarterly — competitive intel especially goes stale fast
- If you haven't referenced a file in 6 months, consider archiving it
- Always include a `Last updated` date so you know when to refresh
